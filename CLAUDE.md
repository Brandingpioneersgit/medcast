# MedCasts

## Architecture
- **Stack**: Next.js 15 (App Router) + PostgreSQL 16 + Drizzle ORM + Tailwind CSS + next-intl
- **Entry**: `src/app/[locale]/layout.tsx` (locale-aware root with RTL support)
- **DB Schema**: `src/lib/db/schema.ts` — 30+ tables (hospitals, doctors, treatments, specialties, conditions, packages, leads, referrals, translations, FAQs, blog)
- **Queries**: `src/lib/db/queries.ts` — all DB access functions
- **i18n**: 8 locales (en, ar, ru, fr, pt, bn, tr, hi), RTL for Arabic, content translations via DB `translations` table
- **API**: `/api/v1/inquiry` (lead capture), `/api/v1/search` (global search)
- **Deploy target**: 2 VPS (US + India) + Cloudflare CDN

## Database Setup (local dev)
- `npm run db:start` — starts embedded PostgreSQL on port 5432
- `npm run db:migrate` — pushes Drizzle schema (needs DATABASE_URL env)
- `npm run db:seed` — seeds 6 hospitals, 10 doctors, 15 treatments, 10 specialties, 8 conditions, testimonials, accreditations
- Data stored in `.pgdata/` (gitignored)

## Technical Debt
- Locale message files (ru/fr/pt/bn/tr/hi) are English copies — UI strings still English; content translations happen via DB `translations` table populated by AI translator
- `any` types on some Drizzle relation iterations
- Upload/email providers default to local fallback — swap to Cloudinary/Resend when env vars set
- Embedded local PG now mirrored to Supabase (prod `DATABASE_URL` in `.env.local`); local still used for import jobs before syncing up
- Need to run `db:migrate` after schema additions: appointments, patient_reviews, background_jobs, doctors.cal_url
- Job worker: wired to QStash scheduled calls (`scripts/setup-qstash.ts` registers `*/5 * * * *`). Bearer `JOBS_TOKEN` still works for manual runs.
- Rate limiter is in-memory (process-local) — switch to Redis/KV once multi-instance.
- Currency switcher reloads the page on change (simple approach) — upgrade to route-refresh once we have a server-action route.
- Service worker offline cache covers shell only; tune per-route strategies when production traffic grows.
- `/portal/[code]` currently looks up appointments only; extend to referral codes + inquiry tracking when portal grows.
